\# Jira SOC Ticketing Workflow Mini-Lab



\## Overview



This mini-lab demonstrates how Security Operations Center (SOC) analysts manage and document security incidents using a ticketing system. In this exercise, a simulated security alert was investigated and tracked using Jira to replicate a real SOC incident management workflow.



Ticketing systems are a core component of SOC operations because they allow analysts to track alerts, document investigation steps, escalate incidents when required, and maintain an audit trail of actions taken.



---



\# Lab Objectives



\- Understand the SOC incident management workflow

\- Create and manage security incident tickets

\- Document investigation steps inside a ticket

\- Simulate the SOC ticket lifecycle from alert to closure

\- Produce investigation documentation aligned with real SOC workflows



---



\# Tools Used



\- Jira (Ticketing / Case Management System)

\- Splunk (Alert Source Simulation)

\- Linux Authentication Logs (/var/log/auth.log)



---

A typical SOC incident lifecycle was simulated during this lab:



SIEM Alert Generated

Priority Level:



Medium



Label:



ssh-bruteforce



The ticket represents the initial alert generated by the SIEM platform.



---



\# Investigation Process



The following steps were documented inside the Jira ticket:



1\. Reviewed Linux authentication logs from `/var/log/auth.log`

2\. Identified multiple failed SSH login attempts targeting the account \*\*socadmin\*\*

3\. Total failed attempts observed: \*\*5\*\*

4\. Source IP address identified as \*\*192.168.1.15\*\*



---



\# Analysis



The source IP address was traced to an internal host within the lab environment.



No additional suspicious behaviour was observed after the failed authentication attempts.



A successful login was recorded shortly after the failed attempts.



This behaviour indicates the activity was likely caused by a legitimate user entering an incorrect password multiple times.



---



\# Actions Taken



\- Verified authentication logs

\- Confirmed successful login event

\- Correlated failed login attempts with internal host activity

\- Documented findings within the Jira ticket



No containment or escalation actions were required.



---



\# Final Conclusion



The activity was determined to be a \*\*false positive caused by user authentication error\*\*.



No indicators of compromise were identified.



The investigation was completed and the incident ticket was closed.
